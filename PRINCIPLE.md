# PRINCIPLE

这个文件记录本工程在协作、调试、仿真过程中已经被验证过的工作守则。目标不是写空泛原则，而是把踩过坑之后确认有效的做法固定下来，作为后续工作的默认约束。

## 1. 仿真执行原则

### 1.1 分阶段执行，不直接长跑整套流程

模块仿真默认按下面顺序执行：

1. 编译 `xvlog`
2. 展开 `xelab`
3. 单独运行 `xsim`

原因：

- 可以快速区分问题是在语法、展开还是运行阶段
- 不会把不同阶段的问题混在一起
- 更容易定位是 DUT、testbench、还是工具环境问题

### 1.2 仿真必须设置短超时

默认不允许无上限等待仿真结束。

要求：

- 单次仿真先用短上限
- 超时后立即停止，不继续盲等
- 超时本身视为一个需要定位的问题，而不是“再等等”

原因：

- 避免一次误判浪费数分钟甚至更久
- 很多卡死本质上是 testbench 等待条件永远不满足

### 1.3 超时后先读日志，再决定下一步

如果仿真超时，下一步不是重跑，而是：

1. 停掉残留 `xsim/xsimk`
2. 读取当前日志
3. 判断卡在哪个阶段、哪个场景、哪个等待条件
4. 再修改代码或 testbench

禁止做法：

- 在没有新信息的前提下重复整套重跑
- 用“多等等看”代替定位

### 1.4 testbench 必须使用有界等待

testbench 中所有 `wait/while` 式等待都应尽量带上边界：

- 最大周期数
- 超时后的明确报错
- 报错时打印关键状态

原因：

- 防止静默卡死
- 出错时直接给出定位信息

### 1.5 先验证单模块，再做回归

修改某个底层模块后：

1. 先让该模块自己的 testbench 通过
2. 再跑依赖它的上层模块回归

原因：

- 避免在上层复杂环境里盲猜底层问题
- 防止“修好了局部，却破坏了集成”

## 2. Testbench 设计原则

### 2.1 testbench 不依赖 DUT 私有内部名

testbench 默认只能依赖：

- DUT 对外端口
- 明确承诺的参数
- 明确约定可观察的层次信号

默认禁止：

- 直接访问 DUT 内部私有寄存器
- 依赖未公开的 `*_int`、临时信号、内部状态名

原因：

- 内部名一改，testbench 就会在 elaboration 阶段直接失效
- 模块接口才是稳定契约

### 2.2 样例必须覆盖正常路径和错误路径

每个模块级 testbench 至少要有：

- 基本正常路径
- 一个或多个边界条件
- 错误或非法输入路径

原因：

- 只测 happy path 不能说明模块可用

### 2.3 检查不能只看“有没有跑完”

testbench 应优先做明确校验，例如：

- 数据逐项比对
- 计数比对
- 握手行为检查
- 错误路径检查

禁止把“仿真跑到了 `$finish`”当作通过标准。

## 3. 仿真模型原则

### 3.1 仿真 fallback 以稳定和可验证为先

如果某个模块在非综合仿真环境下需要 fallback 模型：

- 优先保证功能语义正确
- 优先保证仿真可收敛、可定位
- 不盲目追求和综合实现逐门级等价

原因：

- fallback 的职责首先是支撑模块级功能验证
- 不稳定的“近似真实实现”比稳定的行为模型更糟

### 3.2 改底层仿真模型后必须回归上层

例如 FIFO、CDC、buffer 这类基础模块一旦修改：

- 必须回归依赖它们的模块级 testbench

原因：

- 基础模块改动的影响范围通常大于表面看起来的范围

## 4. 文档同步原则

### 4.1 RTL 中文注释必须随修改同步维护

凡是修改、重写、重构 RTL 模块时：

- 不允许把原先已经存在且仍然有效的中文注释删掉后不补回
- 新增的关键参数、端口、内部状态、门控条件、特殊约束，必须补上中文注释
- 注释要尽量贴近对象本身，优先写在对应参数、信号、端口旁边或上方
- 如果旧注释已与代码行为不一致，必须在同一次修改中一起修正

重点适用对象：

- FIFO/XPM 参数和端口
- CDC 信号与跨时钟握手
- 状态机状态与状态转移条件
- 回压、阈值、计数器、异常检测逻辑

原因：

- 这类模块的正确性依赖设计意图，单看代码很容易误读
- 重构时最容易先改逻辑、后忘注释，最终让代码 review 失去依据

### 4.2 每个 testbench 必须有同名前缀 `.md`

例如：

- `tb_xxx.sv`
- `tb_xxx.md`

`.md` 至少要包含：

- 模块做什么
- 测了哪些样例
- 为什么这样设计样例
- 最近一次仿真结果

### 4.3 改 testbench 时同步改文档

如果修改了：

- 测试样例
- 检查逻辑
- 当前结论
- 最近一次结果

则必须同步更新对应 `.md`。

### 4.4 文档状态必须以当前工作区实测为准

如果别处说“已经通过”，但当前工作区实测没有通过：

- 以当前工作区实际结果为准
- 文档必须反映当前真实状态

### 4.5 动态状态表只能根据真实证据刷新

`module_simulation.md` 中的动态状态表只能基于下面这些真实证据更新：

- `rtl/` 下真实存在的模块文件
- `rtl/sim/` 下真实存在的 testbench 和对应 `.md`
- `sim_out/` 下真实存在的日志文件
- 日志中真实出现的通过或失败结果

禁止做法：

- 因为“理论上应该有”就把某个模块标记为已验证
- 因为“之前某次对话里说过”就把结果写成通过
- 在没有实际日志时写入某个 testbench 版本已经仿真

如果缺少其中任一证据：

- 表格必须保守显示为 `无`、`未验证` 或 `未知`
- 不允许猜测性补全

### 4.6 面向用户的文档与状态表默认使用中文

凡是直接给用户阅读和维护的内容，默认使用中文书写，包括：

- `docs/` 下的说明文档
- `rtl/sim/` 下与 testbench 配套的 `.md`
- `module_simulation.md` 中的状态表
- 脚本生成到文档里的标题、列名、状态词和备注

禁止做法：

- 生成中文文档时夹杂英文列名、英文状态词、英文备注
- 只因为脚本实现方便，就把最终展示内容写成英文

例外情况：

- 工具命令名、文件名、模块名、testbench 名称保留原始英文标识
- 代码内部变量名和脚本内部实现可继续使用英文，但最终输出给用户看的文本必须是中文

## 5. 工具使用原则

### 5.1 先保留日志，再看是否需要 GUI/波形

默认先拿：

- 编译日志
- 展开日志
- 仿真日志

波形用于进一步定位，不替代日志。

### 5.2 面向当前环境选择波形格式

当前环境下：

- `xsim -gui` 适合看 `.wdb`
- `gtkwave` 适合看 `.vcd`

不要混用查看工具和波形格式。

## 6. 当前特别约束

从现在开始，遇到仿真问题时默认执行以下流程：

1. 先编译
2. 再展开
3. 最后单独跑仿真
4. 仿真设置短上限
5. 一旦超时，先读日志定位
6. 不再盲等整套脚本
7. 修完单模块后，补跑相关回归
8. 最后同步更新对应 `.md`
