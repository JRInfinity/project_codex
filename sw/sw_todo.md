# 软件待办

本文档把当前的软件落地方向整理成可逐步执行的计划。

当前目标工作流：

1. 板子通过 SD 卡启动。
2. 自动加载 `FSBL + bitstream + PS 应用程序`。
3. PS 应用程序常驻运行，等待 PC 发来任务。
4. PC 通过一根连线把输入 `.bin` 图像发到板子。
5. PS 将图像写入 DDR，并启动 `image_geo_top`。
6. 处理完成后，PS 立即把输出像素数据回传给 PC。
7. PC 接收结果并显示图像。

## 阶段 0：先把边界定清楚

目标：
- 明确“启动介质”和“运行时传输介质”是两件事。

决定：
- SD 卡只负责启动和部署程序。
- 运行时图像传输通过一根实时连线完成。
- 第一版先用 `UART`，因为最容易最快打通闭环。
- 如果后续吞吐量成为瓶颈，再把同一套协议迁移到以太网。

完成标准：
- 正常运行时不再依赖 SD 卡逐张读写图像。
- 板子上电后可以持续处理多张图，不需要每次重启。

## 阶段 1：定义运行时通信协议

目标：
- 定义一套 PC 和板子之间的小型固定协议。

板端职责：
- 等待请求包。
- 接收图像参数和图像负载。
- 将负载写入 DDR。
- 启动硬件处理。
- 回传状态和输出图像负载。

PC 端职责：
- 打开输入 `.bin`。
- 发送参数和图像字节流。
- 等待处理结果。
- 保存并显示返回图像。

协议 v1 请求头字段：
- `magic`
- `version`
- `command`
- `src_w`
- `src_h`
- `src_stride`
- `dst_w`
- `dst_h`
- `dst_stride`
- `rot_sin_q16`
- `rot_cos_q16`
- `prefetch_enable`
- `payload_bytes`

协议 v1 响应头字段：
- `magic`
- `version`
- `status`
- `output_bytes`
- `hw_status_reg`
- `cache_reads`
- `cache_misses`
- `cache_prefetches`
- `cache_hits`
- `payload_data`

计划产物：
- `sw/include/image_geo_protocol.h`
- 在 `sw/README_PYNQ_Z2.md` 或相关文档中补充协议说明

完成标准：
- 协议字段稳定到足以让板端 C 和 PC 端 Python 同时开始实现。

## 阶段 2：板端常驻服务程序

目标：
- 用“等待 PC 数据包”的主循环替换“从 SD 读文件”的主流程。

主循环形态：

```c
while (1) {
    接收请求头();
    接收图像负载到DDR();
    刷新缓存();
    配置image_geo寄存器();
    启动IP并等待完成();
    失效输出缓存();
    发送响应头();
    回传输出负载();
}
```

板端需要的模块：
- `main.c`
- `image_geo_regs.h`
- `image_geo_driver.[ch]`
- `transport_uart.[ch]`
- `image_geo_protocol.h`

实现要求：
- 第一版先用轮询，不用中断。
- 源图和目标图都使用固定 DDR 区域。
- 校验 `payload_bytes == src_stride * src_h`。
- 如果请求超出预留 DDR 缓冲区范围，直接拒绝。

完成标准：
- 板子无需重启即可连续处理多次请求。

## 阶段 3：PC 端 Python 工具

目标：

PC 端建议产物：

完成标准：

## 阶段 4：SD 启动打包

目标：
- 让板子上电后自动进入常驻服务程序。

需要的交付物：
- `BOOT.BIN`
- 视启动流而定的辅助启动文件

打包内容：
- FSBL
- PL bitstream
- PS 应用 ELF

完成标准：
- 板子上电后直接进入“等待 PC 请求”的状态。

## 阶段 5：调试与验证

目标：
- 让故障更容易定位，不要把问题混在一起。

检查项：
- 板端在处理前打印收到的协议头字段
- 板端在完成后打印硬件状态寄存器
- PC 端在显示前校验回包长度
- 先跑小图：`64x64`、`128x128`
- 小图稳定后再放大到 `640x480`

建议的初始测试：
1. `checkerboard` 恒等变换
2. `horizontal_gradient` 恒等变换
3. `90 degree` 旋转

完成标准：
- 小图闭环稳定。
- 恒等变换模式下返回结果与输入一致。

## 阶段 6：性能升级路径

目标：
- 在 UART 版本跑通后，为更高吞吐量做准备。

后续方向：
- 将传输层从 UART 切换到 Ethernet
- 协议字段尽量保持不变
- 如有需要再增加分块传输和 CRC

完成标准：
- 先保证 UART 版本正确可用。
- 将来升级的原因是速度，而不是正确性问题。
