# 代码学习指南

## 这份指南是干什么的
这份文档不是介绍某一个模块，而是教你如何系统地阅读当前工程里的代码和我补充的说明文档。

你现在面对的不是一两个孤立文件，而是一条完整的数据通路和一套配套工具：
- 上层控制
- DDR 读写
- 跨时钟域传输
- FIFO 缓冲
- 像素拆包
- 缩放核心
- 行缓存与行输出
- 仿真脚本和说明文档

如果直接从某个大文件开始硬啃，容易出现两个问题：
- 只看懂局部语法，看不懂整体数据流
- 看到很多状态机和握手信号，但不知道它们为什么存在

所以最有效的方法不是“按文件名一个个顺序读完”，而是：
1. 先建立全局图
2. 再按数据流方向看模块
3. 再看控制流
4. 最后看脚本、仿真和文档维护方式

这份指南就是按这个顺序展开。

## 二、你应该先建立什么全局认识

先记住一句话：

**这个工程的主线，是把 DDR 里的一段图像数据读出来，跨时钟送到核心逻辑，拆成像素流，做几何/缩放处理，再准备写回。**

从“系统行为”角度，你可以把它拆成下面几个阶段：

1. 控制面发命令
2. 读引擎去 DDR 把数据搬出来
3. 读回来的总线数据先进入 FIFO
4. FIFO 里的整 word 数据被拆成逐像素数据
5. 缩放/几何核心消费这些像素
6. 行缓存与行输出模块把结果组织好
7. 写引擎把结果写回 DDR

你只要一直带着这条主线去读，很多状态机就不再是“杂乱寄存器”，而是某个阶段的实现细节。

## 四、每次看一个模块，应该按什么套路看

这是最重要的“阅读动作模板”。

以后你看每个模块，都按下面顺序。

### 第 1 步：先看模块职责注释
我已经在很多文件开头加了“模块职责”注释。

你先只看这里，先回答：
- 这个模块是做什么的
- 它在整条链路里的位置是什么
- 它的输入输出分别代表什么抽象层级

比如：
- `axi_burst_reader` 的输入是“读命令”，输出是“整 word + 状态”
- `pixel_unpacker` 的输入是“整 word”，输出是“逐像素”
- `scale_core_nearest` 的输入不是“图像帧”，而是“行请求/像素请求的交互接口”

如果这一步没搞清楚，后面一定会乱。

### 第 2 步：看参数区
你要特别注意这些参数：
- 数据宽度类：`DATA_W`、`PIXEL_W`
- 地址宽度类：`ADDR_W`
- 深度类：`FIFO_DEPTH_WORDS`、`DEPTH`
- 最大尺寸类：`MAX_SRC_W`、`MAX_DST_W`
- 限制类：`BURST_MAX_LEN`、`MAX_OUTSTANDING_BEATS`

看参数时要问：
- 这是协议要求，还是业务要求？
- 改它会影响哪里？
- 这个模块是不是只支持某些固定取值？

例如：
- `pixel_unpacker` 目前明显只支持 `PIXEL_W == 8`
- `image_geo_top` 目前明确限定 32bit AXI 数据宽度

### 第 3 步：看端口，不要只看名字，要分组
不要把端口当成一长串信号名。

应该先分组：
- 命令输入
- 数据输入
- 数据输出
- 状态输出
- AXI 总线接口
- CDC 接口
- ready/valid 握手接口

例如 `ddr_read_engine`：
- `task_start/task_addr/task_byte_count` 是任务输入
- `task_busy/task_done/task_error` 是任务状态
- `out_data/out_valid/out_ready` 是像素流输出
- `m_axi_rd` 是 AXI 读总线接口

### 第 4 步：看状态机定义
有状态机的模块，先看 `typedef enum`。

你要在纸上或脑子里把每个状态翻译成一句中文：
- `S_IDLE`：等命令
- `S_PREP`：做预计算
- `S_AW`：发写地址
- `S_WDATA`：发写数据
- `S_BRESP`：等写响应

然后再看 `always_comb` 里的状态跳转条件。

这一步你不要急着看寄存器赋值，只看：
- 从哪个状态跳到哪个状态
- 因为什么条件跳

### 第 5 步：最后看 `always_ff`
前面都明白后，再看 `always_ff`。

这一部分你要重点抓三类东西：
- 哪些寄存器在启动时被初始化
- 哪些计数器在推进数据流
- 哪些标志位在表示“事件已经发生”

比如：
- `words_requested_reg`
- `bytes_remaining_reg`
- `reader_done_seen_reg`
- `task_active_reg`

这些寄存器其实就是模块“内部思维过程”的体现。

## 五、看代码时最应该抓住的几个核心概念

下面这些概念贯穿整个工程。你如果把它们吃透，后面会快很多。

### 1. ready/valid 握手
这是本工程最基础的时序交互方式。

你必须牢牢记住：
- `valid` 表示“我现在有东西”
- `ready` 表示“我现在能接收”
- 只有 `valid && ready` 同时为 1，这次传输才真正发生

在代码里常见写法是：
- `fire = valid && ready`

例如：
- `ar_fire`
- `r_fire`
- `w_fire`
- `out_fire`
- `in_fire`

你看到 `*_fire`，就立刻知道：这是“本拍传输成交”。

### 2. 跨时钟域 CDC
本工程里至少有两个时钟域：
- AXI 域
- core 域

所以你必须区分两类信息：

#### 电平量
例如 `busy`
这类信息通常可以用双触发同步过去。

#### 事件量
例如 `done`、`error`、`start`
这类信息不能直接打一拍同步，因为可能丢脉冲。

所以你看到：
- `task_cdc`
- `result_cdc`

它们本质都是在解决：
**“一次性的事件和它携带的多比特载荷，怎样安全跨域”**

### 3. 数据宽度转换
DDR/AXI 总线返回的是总线宽数据，比如 32bit。
但图像处理通常按像素，比如 8bit。

所以必须有：
- 宽数据 -> 窄数据：`pixel_unpacker`
- 窄数据 -> 宽数据：`ddr_write_engine`

这两个模块是读写链路里最值得对照学习的一对。

### 4. 4KB 边界限制
AXI INCR burst 不能跨 4KB 边界。

所以读写引擎都要做一件事：
- 明明用户给的是“一大段地址范围”
- 但内部必须拆成若干个合法 burst

这也是 `ddr_axi_pkg.sv` 里那些辅助函数存在的意义。

### 5. “控制”和“执行”分离
本工程不是所有逻辑都堆在一个大模块里。

一个典型分层是：
- 控制器：决定谁该动，什么时候动
- 执行器：专注做某一类具体工作

例子：
- `scaler_ctrl` 是控制器
- `scale_core_nearest` 是执行器
- `axi_burst_reader` 是执行器
- `pixel_unpacker` 是执行器

你读代码时要一直问：
- 这个模块是在“做决定”，还是在“干活”

## 七点五、怎么阅读寄存器文档和顶层控制面

这一节专门对应顶层控制类文件，尤其是：
- [image_geo_top.sv](C:\Users\huawei\Desktop\project_codex\rtl\image_geo_top.sv)
- [image_geo_top_说明文档.md](C:\Users\huawei\Desktop\project_codex\docs\code_docs\image_geo_top_说明文档.md)

很多人第一次看顶层控制面时，会陷入两个误区：
- 只会背寄存器地址，但不知道寄存器和内部模块的关系
- 只会看 RTL 里的 `case (addr)`，但不知道软件该怎么配

所以你要分成三层来看。

### 第一层：先把寄存器当“软件接口”

先不要急着看 RTL。

你先把寄存器当成“软件能碰到的唯一入口”，回答下面几个问题：
- 软件一共能写哪些寄存器
- 哪些寄存器是配置参数
- 哪些寄存器是控制命令
- 哪些寄存器是状态反馈

以 `image_geo_top` 为例，可以先粗分成三类：

#### 1. 控制寄存器
- `CTRL`

作用：
- 启动任务
- 控制是否产生中断

#### 2. 配置寄存器
- `SRC_BASE_ADDR`
- `DST_BASE_ADDR`
- `SRC_STRIDE`
- `DST_STRIDE`
- `SRC_SIZE`
- `DST_SIZE`

作用：
- 描述输入图像和输出图像在内存中的布局
- 描述缩放任务的源尺寸和目标尺寸

#### 3. 状态寄存器
- `STATUS`

作用：
- 看整体是否忙
- 看是否完成
- 看是否出错
- 看读写引擎是否忙

如果你先把这三类分清，后面看代码会轻松很多。

### 第二层：再把寄存器和内部信号一一对上

这一层是关键。

你必须学会问：
- 这个寄存器最终写进哪个寄存器变量
- 这个寄存器变量又被哪个子模块使用

比如在 `image_geo_top` 里：

#### `SRC_BASE_ADDR`
软件写入：
- `SRC_BASE_ADDR`

内部寄存器变量：
- `reg_src_base_addr`

最终流向：
- 送给 `scaler_ctrl`

`scaler_ctrl` 再用它去计算：
- 每一条源图行的读取地址

所以你要理解：

**`SRC_BASE_ADDR` 不是“读引擎直接马上访问的地址”，而是“整个任务的源图基地址”。**

#### `DST_BASE_ADDR`
软件写入：
- `DST_BASE_ADDR`

内部寄存器变量：
- `reg_dst_base_addr`

最终流向：
- 送给 `scaler_ctrl`

`scaler_ctrl` 再用它去生成：
- 每一条目标行的写回地址

#### `SRC_SIZE`
软件写入：
- `src_w`
- `src_h`

内部寄存器变量：
- `reg_src_w`
- `reg_src_h`

最终流向：
- 送给 `scale_core_nearest`
- 也送给 `scaler_ctrl`

用途：
- 让控制器知道源图有多大
- 让缩放核心知道坐标映射边界

#### `CTRL.start`
软件写入：
- `CTRL.bit0 = 1`

内部动作：
- 生成 `start_pulse_reg`

最终流向：
- 送给 `scaler_ctrl.start`

这意味着：

**顶层控制寄存器不直接控制读引擎或写引擎，它是先启动总控制器，再由总控制器去调度整条链路。**

这点一定要看懂。

### 第三层：再理解寄存器的“行为语义”

很多寄存器不是“单纯存一个数”，而是带行为的。

你要特别注意三种寄存器位：

#### 1. 脉冲型控制位
例如：
- `CTRL.start`

它的特点：
- 写 1 触发一个动作
- 不是长期保持状态
- 通常读回没有意义，甚至固定读回 0

所以不要把它理解成：
- “start=1 表示一直在开始”

真正含义是：
- “软件写 1 的那个瞬间，硬件拉起一次启动脉冲”

#### 2. sticky 状态位
例如：
- `STATUS.done_sticky`
- `STATUS.error_sticky`

它们的特点：
- 某个事件发生后被置位
- 不会自动马上清掉
- 软件可以稍后再读

你可以把它理解成：
- “事件发生过的痕迹”

这对于软件很重要，因为软件不一定恰好在那个单拍事件出现时正在读寄存器。

#### 3. W1C 位
例如：
- `STATUS.bit1`
- `STATUS.bit2`

W1C 不是普通写寄存器。

它的语义是：
- 写 1 清零
- 写 0 不变

你以后看到任何寄存器文档里写 `W1C`，都要自动想到这一点。

---

### 第四层：寄存器文档应该怎么顺着读

我建议你每次按这个顺序读寄存器文档：

1. 先看总表
2. 再看每个寄存器的位定义
3. 再看写入行为细节
4. 最后看示例

不要一上来就看示例。

正确方式是：

#### 第一步：看总表
先知道：
- 有哪些寄存器
- 哪些是配置，哪些是控制，哪些是状态

#### 第二步：看位定义
明确每一位是做什么的。

#### 第三步：看编码规则
尤其注意：
- 宽高拼接顺序
- 地址单位是不是字节
- stride 单位是不是字节

#### 第四步：看写入行为
这里很容易踩坑。

例如：
- `start` 是写 1 触发
- `STATUS` 的 sticky 位是写 1 清零
- 配置寄存器受 `WSTRB` 控制，允许按字节更新

#### 第五步：最后看示例
示例不是替代理解，而是验证你已经理解。

### 第五层：怎么把寄存器配置顺序和硬件行为联系起来

你以后看到软件配置序列，不要只背地址和值。

要在脑子里同步想这条链路：

1. 写 `SRC_BASE_ADDR`
2. 写 `DST_BASE_ADDR`
3. 写 `SRC_STRIDE`
4. 写 `DST_STRIDE`
5. 写 `SRC_SIZE`
6. 写 `DST_SIZE`
7. 清 `STATUS` sticky
8. 写 `CTRL.start`

这在硬件里对应的是：

1. 顶层保存源图/目标图配置
2. `scaler_ctrl` 将来会使用这些配置
3. 当 `CTRL.start` 写入后，产生 `start_pulse_reg`
4. `scaler_ctrl` 被启动
5. `scaler_ctrl` 决定何时去读源图、何时启动核心、何时写回
6. 完成后 `done_sticky` 置位，出错则 `error_sticky` 置位
7. 若 `irq_en=1`，则 `irq` 拉高

这就是你从“软件写寄存器”到“硬件整条链路启动”的完整理解。

### 第六层：看顶层控制面时，你最应该问的几个问题

拿着寄存器文档和 `image_geo_top.sv`，你应该反复问：

#### 1. 哪些寄存器只是配置？
比如：
- `SRC_BASE_ADDR`
- `DST_BASE_ADDR`
- `SRC_STRIDE`
- `DST_STRIDE`
- `SRC_SIZE`
- `DST_SIZE`

这些寄存器只是保存参数，不会因为你写了它们就立即启动任务。

#### 2. 哪个寄存器真正触发任务？
是：
- `CTRL.start`

#### 3. 任务完成后软件靠什么知道？
有三种办法：
- 读 `STATUS.done_sticky`
- 读 `STATUS.error_sticky`
- 等 `irq`

#### 4. sticky 位为什么有必要？
因为：
- `ctrl_done` 和 `ctrl_error` 在硬件内部通常是脉冲
- 软件轮询可能错过瞬间脉冲
- sticky 位能把“发生过”保存下来

#### 5. 哪些状态是“当前状态”，哪些是“历史事件”？

当前状态：
- `STATUS.ctrl_busy`
- `STATUS.read_busy`
- `STATUS.write_busy`

历史事件：
- `STATUS.done_sticky`
- `STATUS.error_sticky`

这两类语义不能混。

### 第七层：你读寄存器文档时最容易踩的坑

#### 1. 把 `start` 当成状态位
错。

它是触发位，不是持续状态位。

#### 2. 把 `done_sticky` 当成“当前一定已经空闲”
不完全对。

通常完成后会空闲，但 sticky 位本质上表示：
- “完成事件曾经发生过”

它不是忙闲状态本身。

#### 3. 忽略地址和 stride 的单位
这里单位是：
- 字节

不是：
- 像素
- word

#### 4. 忽略 `SRC_SIZE` / `DST_SIZE` 的拼接方式
必须记住：
- 低 16 位是宽
- 高 16 位是高

#### 5. 看到 `STATUS` 可写就以为它是普通 RW
不是。

它是：
- 主要用于读
- 其中 sticky 位支持 `W1C`

## 八、你应该做什么笔记

如果你真的要系统学习，建议你自己额外记三类笔记。

### 1. 模块关系笔记
每个模块只写三件事：
- 输入是什么
- 输出是什么
- 它依赖谁

例如：
- `pixel_unpacker`
- 输入：FIFO word、reader 状态
- 输出：像素流、done/error
- 依赖：`ddr_read_engine` 提供上游数据

### 2. 状态机笔记
对每个状态机写：
- 状态名
- 进入条件
- 退出条件
- 该状态干什么

这会极大提高你读 RTL 的速度。

### 3. 信号语义笔记
有些信号看名字容易混，但意义非常关键。

比如：
- `busy`：模块当前占用中
- `done`：本次任务成功完成
- `error`：本次任务发生错误
- `valid`：当前拍提供有效信息
- `ready`：当前拍可以接收信息
- `fire`：当前拍交易真正发生

你最好把这些术语固定下来。

## 九、初学时最容易犯的错误

### 1. 一上来就想把所有细节看完
这是最常见的错误。

RTL 不适合“逐字符通读”。
你应该先看层次和职责，再看细节。

### 2. 只盯寄存器赋值，不看状态转移
如果你只看 `always_ff` 里的赋值，很容易淹没在细节里。

正确顺序是：
- 先看状态机
- 再看时序赋值

### 3. 忽略时钟域
这是很危险的。

你一旦分不清：
- 哪段逻辑在 `axi_clk`
- 哪段逻辑在 `core_clk`

就会误解很多 CDC 逻辑存在的必要性。

### 4. 把“数据到了”误认为“任务完成”
尤其是在 `pixel_unpacker`、`row_out_buffer`、写回链路里，这个误解很常见。

系统里很多 done 的语义是：
- 数据已经被下游真正消费

不是：
- 数据刚刚出现在某个寄存器里

### 5. 忽略错误路径
很多人只看 happy path。

但这个工程里：
- `RRESP` 错误
- `RLAST` 不匹配
- FIFO overflow/underflow
- 参数为 0

这些错误路径都很关键。

你真正看懂一个模块，必须同时看：
- 正常路径
- 错误路径
