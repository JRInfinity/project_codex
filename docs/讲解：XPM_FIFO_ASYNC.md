# 1. 先建立整体理解：这是什么 FIFO？

`xpm_fifo_async` 是 **异步 FIFO**，意思是：

- **写端** 用一个时钟域：`wr_clk`
- **读端** 用另一个时钟域：`rd_clk`

也就是它本质上是一个 **跨时钟域缓冲器**。

你可以把它理解成中间一个“仓库”：

- 写侧不断往里面放数据
- 读侧按自己的时钟把数据取出来
- FIFO 自动帮你处理跨时钟域同步问题

所以它的端口天然分成两堆：

- **写域端口**：跟 `wr_clk` 相关
- **读域端口**：跟 `rd_clk` 相关

------

# 2. 最核心、最常用的一组端口

这是你平时最先要会用的一组。

------

## 2.1 `wr_clk`

- **方向**：Input
- **作用**：写时钟
- **所属时钟域**：写域

### 它是干什么的？

写端所有操作都在 `wr_clk` 下进行：

- `wr_en` 是否有效
- `din` 被什么时候写进去
- `full / almost_full / prog_full / wr_data_count / overflow / wr_ack / wr_rst_busy`
   这些写侧相关信号，通常都在这个域里产生或更新

### 直白理解

谁往 FIFO 里塞数据，谁就跟着 `wr_clk` 走。

------

## 2.2 `rd_clk`

- **方向**：Input
- **作用**：读时钟
- **所属时钟域**：读域

### 它是干什么的？

读端所有操作都在 `rd_clk` 下进行：

- `rd_en` 是否有效
- `dout` 在什么时候输出
- `empty / almost_empty / prog_empty / data_valid / rd_data_count / underflow / rd_rst_busy`
   这些读侧相关信号，通常都在这个域里变化

### 直白理解

谁从 FIFO 里拿数据，谁就跟着 `rd_clk` 走。

------

## 2.3 `rst`

- **方向**：Input
- **作用**：复位 FIFO
- **图里说明**：**必须同步到 `wr_clk`**

### 这一点很关键

虽然它叫异步 FIFO，但这个 `rst` **不是随便乱来的异步释放复位**。
 XPM 文档里强调的是：

- 施加复位时，时钟可能不稳定
- 但**释放复位时，时钟必须已经稳定**
- 而且这个复位要满足模板要求，通常认为它是**相对写时钟同步的复位控制输入**

### 你实际设计里怎么理解？

最稳妥的做法是：

- 用系统稳定后的复位信号
- 按官方模板要求去驱动 `rst`
- 不要在 `wr_rst_busy` 或 `rd_rst_busy` 高的时候就立刻开始读写

### 直白理解

`rst=1` 时，相当于告诉 FIFO：

> “现在清空状态，写指针读指针都重新来。”

------

## 2.4 `din`

- **方向**：Input
- **宽度**：`WRITE_DATA_WIDTH`
- **所属时钟域**：写域

### 它是干什么的？

这是**写入 FIFO 的数据输入总线**。

当某个 `wr_clk` 上升沿到来时，如果：

- `wr_en = 1`
- 且 FIFO **不满**

那么这拍 `din` 上的数据就会被写进 FIFO。

### 直白理解

`din` 就是“准备放进仓库的货”。

------

## 2.5 `wr_en`

- **方向**：Input
- **宽度**：1
- **所属时钟域**：写域
- **有效电平**：高有效

### 它是干什么的？

写使能。
 当 FIFO 不满时，`wr_en=1` 表示“这一拍要写数据”。

### 必须注意

图里专门写了：

- 当 `rst` 为高时
- 或 `wr_rst_busy` 为高时

**必须让 `wr_en` 保持低**

也就是在写侧复位忙的时候，不准写。

### 常见写法

```
if (!full && !wr_rst_busy)
    wr_en <= 1'b1;
else
    wr_en <= 1'b0;
```

### 直白理解

`wr_en` 就像“入库按钮”。

------

## 2.6 `full`

- **方向**：Output
- **宽度**：1
- **所属时钟域**：写域
- **高有效**

### 它是干什么的？

FIFO 满标志。

- `full=1`：不能再写了
- `full=0`：还可以继续写

### 注意

当 FIFO 满时，如果你还拉高 `wr_en`，这次写请求会被拒绝，不会把内容写坏。
 这时往往还会伴随 `overflow` 置位。

### 直白理解

仓库满了，门口挂上“停止入库”。

------

## 2.7 `dout`

- **方向**：Output
- **宽度**：`READ_DATA_WIDTH`
- **所属时钟域**：读域

### 它是干什么的？

这是从 FIFO 读出来的数据。

### 但这里要注意一个点

`dout` **什么时候有效**，不能只看 `dout` 本身，要配合：

- `rd_en`
- `empty`
- `data_valid`
- 以及参数 `READ_MODE`

一起看。

通常最稳妥的理解是：

- 当读操作成功后，`dout` 上会给出读出的那个字
- 最好用 `data_valid` 判断当前 `dout` 是否真的有效

### 直白理解

`dout` 就是“从仓库取出来的货”。

------

## 2.8 `rd_en`

- **方向**：Input
- **宽度**：1
- **所属时钟域**：读域
- **高有效**

### 它是干什么的？

读使能。
 当 FIFO 不空时，`rd_en=1` 表示“这一拍要读一个数据”。

### 必须注意

图里也强调了：

- 当 `rd_rst_busy=1` 时，`rd_en` 必须保持低

也就是读侧复位忙的时候不能读。

### 常见写法

```
if (!empty && !rd_rst_busy)
    rd_en <= 1'b1;
else
    rd_en <= 1'b0;
```

### 直白理解

`rd_en` 就像“出库按钮”。

------

## 2.9 `empty`

- **方向**：Output
- **宽度**：1
- **所属时钟域**：读域
- **高有效**

### 它是干什么的？

FIFO 空标志。

- `empty=1`：里面没数据，不能读
- `empty=0`：可以读

### 注意

当 `empty=1` 时你还去拉 `rd_en`，这次读请求会被拒绝，FIFO 内容不会被破坏。
 往往还会产生 `underflow`。

### 直白理解

仓库空了，门口挂上“无货可取”。

------

# 3. 很重要但很多人第一次会忽略的端口

------

## 3.1 `wr_rst_busy`

- **方向**：Output
- **宽度**：1
- **所属时钟域**：写域
- **高有效**

### 它是干什么的？

表示 **写侧当前还在复位处理中**。

只要它是 1，就说明写域这边还没完全准备好。

### 设计要求

当它为高时：

- 不要写
- `wr_en` 必须拉低

### 为什么需要它？

因为异步 FIFO 内部有跨域同步电路，复位不是“一拍就彻底结束”的。
 所以即使你外面已经把 `rst` 放开，内部写侧可能还在恢复。

### 直白理解

写侧门卫还在“清场”，先别往里送货。

------

## 3.2 `rd_rst_busy`

- **方向**：Output
- **宽度**：1
- **所属时钟域**：读域
- **高有效**

### 它是干什么的？

表示 **读侧当前还在复位处理中**。

### 设计要求

当它为高时：

- 不要读
- `rd_en` 必须拉低

### 直白理解

读侧门卫还在“整理仓库”，先别取货。

------

## 3.3 `data_valid`

- **方向**：Output
- **宽度**：1
- **所属时钟域**：读域
- **高有效**

### 它是干什么的？

这个信号表示：

> 当前 `dout` 上的数据是有效的。

### 为什么它有用？

因为很多时候你不能只靠 `rd_en` 判断 `dout` 有没有新数据，尤其是在：

- FIFO 读延迟不是 0
- 或不同 `READ_MODE`
- 或你想让下游电路明确知道哪一拍数据可采样

这时候 `data_valid` 非常好用。

### 直白理解

`dout` 是“货物”，`data_valid` 是“出库单盖章：这批货有效”。

------

# 4. 容量/水位相关端口

这些端口不是绝对必须，但工程里非常好用。

------

## 4.1 `almost_full`

- **方向**：Output
- **所属时钟域**：写域

### 它是干什么的？

表示 FIFO **快满了**，但还没到真正 `full`。

一般意思是：

> 再写一两个就要满了。

### 作用

给上游提前刹车，而不是等 `full` 才停。

### 直白理解

仓库还没爆仓，但已经差不多塞满了。

------

## 4.2 `almost_empty`

- **方向**：Output
- **所属时钟域**：读域

### 它是干什么的？

表示 FIFO **快空了**，但还没完全空。

### 作用

给下游提前预警，比如：

- 准备停读
- 准备切换数据源
- 防止马上读空

### 直白理解

仓库里只剩一点货了。

------

## 4.3 `prog_full`

- **方向**：Output
- **所属时钟域**：写域

### 它是干什么的？

**可编程满阈值标志**。
 当 FIFO 中数据量 **大于等于你设置的阈值** 时，它拉高。

### 和 `almost_full` 的区别

- `almost_full`：通常是固定“快满”
- `prog_full`：你可以自己定义阈值

比如你设阈值为 100：

- FIFO 里 >=100 个字，`prog_full=1`

### 作用

很适合做流控。

比如你可以规定：

- 到 80% 容量就让上游暂停

### 直白理解

不是系统说“快满”，而是你自己规定“到这条线就报警”。

------

## 4.4 `prog_empty`

- **方向**：Output
- **所属时钟域**：读域

### 它是干什么的？

**可编程空阈值标志**。
 当 FIFO 里数据量 **小于等于你设置的阈值** 时，它拉高。

### 作用

比如你可以规定：

- 剩下 <=16 个字就提醒下游快没数据了

### 直白理解

你自己画一条“低水位线”，低于它就报警。

------

## 4.5 `wr_data_count`

- **方向**：Output
- **宽度**：`WR_DATA_COUNT_WIDTH`
- **所属时钟域**：写域

### 它是干什么的？

表示从**写侧视角**看到 FIFO 里大概有多少个“写宽度单位”的数据。

### 注意

它不是“全局绝对精确到每一拍”的数学真值，
 因为异步 FIFO 有跨时钟域同步，计数在两个域中看到的值可能有同步延迟。

### 但工程上够不够用？

通常够用，用来：

- 判断 FIFO 大致水位
- 做流控
- 做调试观察

### 直白理解

站在写侧门口看，仓库里大概堆了多少箱。

------

## 4.6 `rd_data_count`

- **方向**：Output
- **宽度**：`RD_DATA_COUNT_WIDTH`
- **所属时钟域**：读域

### 它是干什么的？

表示从**读侧视角**看到 FIFO 里大概还有多少个“读宽度单位”的数据可读。

### 和 `wr_data_count` 的区别

不是一个域，也不一定是同一个单位宽度。
 如果配置了读写宽度比，这两个 count 的含义要按各自宽度理解。

### 直白理解

站在读侧门口看，仓库里大概还剩多少箱可取。

------

# 5. 成功/失败指示端口

这些端口对调试很有帮助。

------

## 5.1 `wr_ack`

- **方向**：Output
- **所属时钟域**：写域
- **高有效**

### 它是干什么的？

表示 **前一个写请求成功了**。

也就是：

- 上一拍 `wr_en=1`
- 且 FIFO 确实接受了写入

那么 `wr_ack` 会给出确认。

### 有什么用？

如果你想严格统计“成功写入了多少次”，`wr_ack` 比单看 `wr_en` 更准。

### 直白理解

仓库管理员回你一句：“这次货我收到了。”

------

## 5.2 `overflow`

- **方向**：Output
- **所属时钟域**：写域
- **高有效**

### 它是干什么的？

表示 **前一个写请求失败了，因为 FIFO 已满**。

即：

- 你请求写
- 但 FIFO 满了
- 这次写没进去

### 重要点

这通常是“告警”，**不是破坏性操作**。
 FIFO 里的已有数据不会因为这次写失败而坏掉。

### 直白理解

你硬要送货，但仓库说：“满了，拒收。”

------

## 5.3 `underflow`

- **方向**：Output
- **所属时钟域**：读域
- **高有效**

### 它是干什么的？

表示 **前一个读请求失败了，因为 FIFO 为空**。

即：

- 你请求读
- 但 FIFO 已经空了
- 这次读没读到有效新数据

### 重要点

这同样一般是“告警”，不破坏 FIFO 内容。

### 直白理解

你去取货，但仓库说：“已经空了，拿不到。”

------

# 6. ECC 相关端口

这些端口只有在启用 ECC 的配置下才有意义。
 如果你当前只是普通图像缓存/跨时钟缓冲，很多时候可以先不管。

------

## 6.1 `sbiterr`

- **方向**：Output
- **所属时钟域**：读域

### 作用

表示 ECC 解码器检测到并**纠正了单比特错误**。

### 直白理解

数据有 1 位出了问题，但系统已经自动修好了。

------

## 6.2 `dbiterr`

- **方向**：Output
- **所属时钟域**：读域

### 作用

表示 ECC 解码器检测到**双比特错误**，而且数据可能已经损坏。

### 直白理解

错误比较严重，系统发现了，但不一定救得回来。

------

## 6.3 `injectsbiterr`

- **方向**：Input
- **所属时钟域**：写域

### 作用

人为注入单比特错误，通常用于 ECC 测试。

### 直白理解

这是“故意制造一个小错误”，看 ECC 能不能抓到并纠正。

------

## 6.4 `injectdbiterr`

- **方向**：Input
- **所属时钟域**：写域

### 作用

人为注入双比特错误，用于 ECC 测试。

### 直白理解

故意制造更严重的错误做测试。

## `sleep`

- **方向**：Input
- **宽度**：1
- **高有效**

### 它是干什么的？

让 memory/FIFO block 进入低功耗模式。

### 什么时候用？

一般只有在你明确做低功耗控制时才会用。
 普通 FPGA 图像处理设计里，常常直接接 `1'b0`。

### 直白理解

告诉 FIFO：“现在先休眠省电。”

# 8. 你现在最该掌握的“最小可用端口集”

如果你现在只是想先把 FIFO 用起来，真正最重要的是这几个：

### 写侧

- `wr_clk`
- `rst`
- `din`
- `wr_en`
- `full`
- `wr_rst_busy`

### 读侧

- `rd_clk`
- `dout`
- `rd_en`
- `empty`
- `rd_rst_busy`
- 最好再加 `data_valid`

这套就已经够你完成：

- 跨时钟域缓存
- 上游写数据
- 下游读数据
- 防止满写/空读
- 正确处理复位

# 9. 一个非常实用的端口使用规则

你以后写控制逻辑时，可以先死记这四条：

### 写入条件

```
wr_en <= write_req && !full && !wr_rst_busy;
```

### 读取条件

```
rd_en <= read_req && !empty && !rd_rst_busy;
```

### 复位忙时禁止操作

- `wr_rst_busy=1` 时不要写
- `rd_rst_busy=1` 时不要读

### 输出数据最好看 `data_valid`

尤其是你后面接流水线模块时，这会让接口更稳。

# 一、先建立整体框架（非常重要）

所有参数其实可以分成 **6大类**：

| 类别        | 作用              |
| ----------- | ----------------- |
| ① 数据结构  | 位宽、深度        |
| ② 行为模式  | 读模式、延迟      |
| ③ 流控/状态 | almost/prog/count |
| ④ 时钟与CDC | 跨时钟可靠性      |
| ⑤ 存储资源  | BRAM / LUT        |
| ⑥ 高级功能  | ECC / 调试        |

# 二、最核心参数（必须搞懂）

------

## 1️⃣ WRITE_DATA_WIDTH

```
写入数据位宽（din）
```

### 关键点

- 范围：1 ~ 4096
- 决定：
  - `din` 位宽
  - FIFO 每个“存储单元”的大小

### ⚠️ 超重要规则

你截图里写了：

👉 写宽和读宽必须满足比例：

```
1:1, 1:2, 1:4, 1:8, 8:1, 4:1, 2:1
```

------

### 举例

| WRITE | READ | 含义           |
| ----- | ---- | -------------- |
| 8     | 8    | 普通 FIFO      |
| 8     | 32   | 4个写合成1个读 |
| 32    | 8    | 1个写拆成4个读 |

------

### 🚀 工程建议（你这个项目）

👉 图像处理（灰度8bit）：

```
WRITE_DATA_WIDTH = 8
READ_DATA_WIDTH  = 8
```

👉 DDR / AXI（通常32/64/128bit）：

```
WRITE = 64
READ  = 64
```

------

## 2️⃣ READ_DATA_WIDTH

```
读取数据位宽（dout）
```

### 核心理解

它和 WRITE_DATA_WIDTH 一起决定：

👉 FIFO 是否做 **宽度转换**

------

### 一个非常重要的概念

如果：

```
WRITE = 8
READ  = 32
```

那么 FIFO 内部行为是：

```
写 4 次 → 才能读 1 次
```

------

## ⚠️ 一个坑（你必须知道）

如果：

```
FIFO_MEMORY_TYPE = "auto"
```

👉 官方要求：

```
WRITE_DATA_WIDTH == READ_DATA_WIDTH
```

否则可能出问题（你截图里写了）

------

------

## 3️⃣ FIFO_WRITE_DEPTH ⭐⭐⭐

```
FIFO 深度（写侧单位）
```

### 含义

👉 FIFO 可以存多少个“写数据单位”

------

### 举例

```
WRITE_DATA_WIDTH = 8
FIFO_WRITE_DEPTH = 1024
```

👉 实际容量：

```
1024 × 8bit = 1KB
```

------

### ⚠️ 必须是 2 的幂

```
16, 32, 64, 128, 256, 512, 1024...
```

------

### 🚀 工程建议

你这个项目（图像处理 + DDR）：

👉 推荐：

```
512 ~ 4096
```

原因：

- AXI 是 burst
- 需要缓冲一段数据

------

------

## 4️⃣ READ_MODE ⭐⭐⭐

```
"std" 或 "fwft"
```

------

### 模式1：std（默认）

👉 标准模式：

```
rd_en 拉高 → 下一拍 dout 才有效
```

------

### 模式2：fwft（强烈推荐）

👉 First Word Fall Through：

```
FIFO 非空 → dout 自动就有数据
```

------

### 对比

| 模式 | 延迟 | 是否需要 rd_en |
| ---- | ---- | -------------- |
| std  | 1拍  | 必须           |
| fwft | 0拍  | 可选           |

------

### 🚀 工程建议（非常关键）

👉 你的项目：

```
READ_MODE = "fwft"
```

原因：

- 流水线更顺
- 少一拍延迟
- 更适合 AXI/视频流

------

------

## 5️⃣ FIFO_READ_LATENCY

```
读路径寄存器级数
```

------

### 含义

👉 dout 前面加几级寄存器

------

### 规则

- std 模式：0~10
- fwft 模式：**必须=0**

------

### 🚀 建议

```
fwft → 0
std  → 1（默认）
```

## 6️⃣ PROG_FULL_THRESH ⭐⭐⭐

```
自定义“快满”阈值
```

------

### 含义

当 FIFO 写入 ≥ 阈值：

```
prog_full = 1
```

------

### 🚀 工程用法

👉 控制 AXI：

```
if (prog_full)
    停止发 burst
```

------

## 7️⃣ PROG_EMPTY_THRESH

```
自定义“快空”阈值
```

------

### 含义

当 FIFO ≤ 阈值：

```
prog_empty = 1
```

------

### 🚀 用途

👉 视频/流处理：

- 防止马上读空
- 提前补数据

------

------

## 8️⃣ WR_DATA_COUNT_WIDTH / RD_DATA_COUNT_WIDTH

```
计数器位宽
```

------

### 公式

```
log2(depth) + 1
```

------

### 举例

```
depth = 1024 → log2=10 → width=11
```

------

### 🚀 用途

- 调试
- 动态流控

## 9️⃣ USE_ADV_FEATURES ⭐⭐⭐

这个是一个“开关集合”，非常重要。

------

### 默认：

```
"0707"
```

------

### 本质

👉 控制哪些端口被启用

------

### 常见位

| bit  | 功能          |
| ---- | ------------- |
| 0    | overflow      |
| 1    | prog_full     |
| 2    | wr_data_count |
| 3    | almost_full   |
| 8    | underflow     |
| 9    | prog_empty    |
| 10   | rd_data_count |
| 11   | almost_empty  |
| 12   | data_valid    |

------

### 🚀 建议（你这个项目）

👉 保持默认：

```
"0707"
```

👉 或者：

```
开启 data_valid（很有用）
```

# 八、总结一句话

👉 所有参数的核心本质只有三件事：

```
1️⃣ 存多少（depth × width）
2️⃣ 怎么读（std / fwft）
3️⃣ 怎么控流（full / prog / count）
```
